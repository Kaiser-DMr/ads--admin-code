# 后台 UI 框架稳定化设计

## 背景

当前投放后台已经完成一轮 Tesla 风格视觉改造，但整体框架层仍存在结构不稳定的问题：

- 二级页面之间的标题、按钮、卡片、表格起始线不完全一致
- 左侧菜单在页面滚动时出现跟随滚动或视觉错觉
- 收起态下侧边栏顶部、菜单、底部用户区在中轴与间距上不够稳定
- 不同页面对顶部留白、操作区、内容区的处理方式不统一

这轮优化的目标是“先稳再美”：先把后台的框架规则统一，彻底收干净错位、滚动、收起等结构问题，再为后续继续增强 Tesla 风格打下稳定基础。

## 目标

1. 建立一套统一的后台框架基线，适用于 Dashboard、Campaigns、Creatives、Reports、Users。
2. 明确滚动责任，让侧边栏、顶部区、内容区的行为一致且可预测。
3. 统一页面内容起始线、垂直节奏和操作区布局，消除不同页面之间的结构漂移。
4. 让收起/展开侧边栏只影响边栏自身，不破坏主内容区的稳定布局。

## 非目标

- 不重做业务逻辑、路由结构、表格数据、弹窗表单逻辑
- 不在这轮新增新的业务组件或功能模块
- 不做激进的视觉风格再设计，只做必要的结构收口与少量视觉统一

## 方案对比

### 方案 A：结构基线优先（采用）
先定义全局布局规则，再把所有页面套到同一骨架上。

优点：
- 最适合解决错位、滚动、收起态不稳定等根问题
- 后续视觉打磨不会反复返工
- 更容易形成可复用的全局 CSS 规则

缺点：
- 第一轮的视觉变化相对克制

### 方案 B：逐页修复
按页面分别处理布局问题。

优点：
- 单页见效快

缺点：
- 容易造成页面之间再次分叉
- 全局一致性最难保证

### 方案 C：纯视觉压层
主要通过 CSS 视觉参数修饰结构问题。

优点：
- 改动快

缺点：
- 无法彻底解决滚动、对齐、收起态等结构层问题

## 采用方案

采用 **方案 A：结构基线优先**。

## 设计

### 1. 整体布局骨架

后台页面划分为两大区域：

- 左侧固定边栏
- 右侧主内容区

右侧主内容区再拆成：

- 顶部全局 Header
- 下方页面内容区

其中：

- 全局 Header 负责承载页面主标题与全局用户信息
- 页面内容区负责承载每个业务页面的操作按钮、卡片、表格、图表等主体内容
- 二级页面不再重复承担“主标题层”的职责，避免双层标题并存

### 2. 侧边栏规则

#### 展开态

- 使用统一展开宽度
- 顶部品牌区、菜单区、底部用户区保持固定垂直分层
- 边栏整体高度锁定为视口高度
- 菜单区占据中间弹性空间

#### 收起态

- 使用统一收起宽度
- 顶部收起按钮、品牌图标、菜单图标、底部用户头像保持同一中轴线
- 收起态只保留必要识别信息，不允许局部元素漂移、重叠、挤压
- 展开/收起切换时，边栏自身变化平滑，但不改变主内容区内边距规则

#### 滚动行为

- 边栏本体固定在视口内
- 边栏外壳不跟随内容区滚动
- 如果菜单项变多，只允许菜单区自身滚动
- 顶部品牌区和底部用户区保持固定

### 3. 顶部 Header 规则

- Header 作为全局信息层，承担页面标题、页面说明、用户信息入口
- 所有页面共用统一的 Header 高度感、水平内边距、圆角、背景与边框层级
- Header 与页面主体内容之间保持固定间距
- Header 文案与页面内容必须共享统一的左侧起始线

本轮优先保证 Header 稳定和对齐，不做激进吸顶交互。若后续需要吸顶，可在结构稳定后单独规划。

### 4. 内容区规则

- 内容区为唯一的主滚动区域
- 内容区左右内边距全局统一
- 页面内部采用统一的 page shell 组织结构
- 页面操作区（page toolbar）与主卡片区（section card）保持一致节奏

统一后的页面层级为：

1. 全局 Header
2. 页面操作区（可为空）
3. 页面主卡片 / 表格 / 图表

### 5. 页面对齐规则

所有页面需要遵守统一内容起始线：

- Header 标题文案
- 页面操作区按钮
- 主卡片或主表格
- 多个内容模块的主边界

这些元素必须沿同一左边界展开，避免出现：

- 页面标题比主卡片更靠左或更靠右
- 操作按钮悬浮在不同页面的不同位置
- 某些页面卡片与标题不共线

### 6. 垂直节奏规则

统一后台垂直间距节奏：

- Header 到页面操作区：固定间距
- 页面操作区到主卡片：固定间距
- 卡片与卡片之间：固定间距
- 表格、Tabs、图表等二级结构跟随统一节奏，而不是按页面单独打补丁

这可以让页面切换时的阅读节奏稳定下来，避免有的页面显得松散、有的页面显得拥挤。

### 7. 页面标题策略

- 全局 Header 是页面的主标题来源
- 二级页面内部不再重复放置同层级标题块
- 页面内部如果仍需说明信息，应降级为操作区说明或局部模块说明，而不是再次出现与 Header 重复的“标题 + 副标题”组合

当前方向下：

- Campaigns、Creatives、Reports、Users 等二级页，应优先保留操作按钮和内容区
- 重复的页内标题块应删除或不再作为主层级存在

### 8. 全局样式沉淀

本轮框架稳定化之后，应将以下内容固化为全局标准：

#### 边栏标准
- 展开宽度
- 收起宽度
- 顶部区布局
- 菜单滚动方式
- 底部用户区位置与对齐方式

#### 主区标准
- Header padding 与层级样式
- 内容区 padding
- Header 与内容区的间距
- 主内容区的最大阅读节奏

#### 页面标准
- `page-shell`
- `page-toolbar`
- `page-section-card`
- 表格容器、卡片头部、分页区的统一边距规则

这些规则应优先落在全局样式文件中，而不是分散在各页面的内联样式或局部补丁里。

## 涉及范围

主要涉及：

- `frontend/src/index.css`
- `frontend/src/layouts/MainLayout.jsx`
- `frontend/src/pages/Dashboard.jsx`
- `frontend/src/pages/Campaigns.jsx`
- `frontend/src/pages/Creatives.jsx`
- `frontend/src/pages/Reports.jsx`
- `frontend/src/pages/Users.jsx`

## 实施原则

1. **最小必要改动**：只收框架，不大改业务实现。
2. **全局优先**：能通过全局布局规则解决的，不做单页补丁。
3. **单一主标题源**：避免 Header 与页内标题重复承担同一职责。
4. **稳定优先于炫技**：先保证结构、滚动、收起、对齐稳定，再做更强的 Tesla 风格表达。

## 验收标准

完成后应满足：

1. 左侧菜单固定，页面滚动时不再跟随主内容区移动。
2. 收起态下边栏顶部、菜单、底部用户区无重叠、错位、漂移。
3. Header、操作按钮、卡片、表格在所有页面上共用统一左起始线。
4. 各页面顶部留白、操作区与主内容区间距一致。
5. 二级页面不再出现与全局 Header 重复的主标题块。
6. 页面切换后，整体观感像同一套后台设计系统，而不是单页拼接。

## 测试策略

- 手动验证展开态与收起态切换
- 手动验证二级页面滚动时边栏行为
- 对比 Dashboard、Campaigns、Creatives、Reports、Users 的标题、按钮、卡片起始线
- 检查不同页面下操作区是否出现多余留白或塌陷
- 执行前端构建，确认布局收口未引入编译错误

## 后续阶段

当这轮“结构稳定化”完成后，下一阶段才适合继续做更明显的 Tesla 风格增强，例如：

- 更统一的卡片层级表达
- 更克制的色彩与高光节奏
- 表格、Tabs、筛选器的高级感统一
- 顶部 Header 的吸顶或轻量动态效果
